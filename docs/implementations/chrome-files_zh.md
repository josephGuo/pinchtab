# Chrome 用户数据目录和配置文件

PinchTab 使用专用的**用户数据目录**（配置文件）来管理 Chrome 实例。本文档解释了这些目录如何解析、如何管理锁，以及如何处理并行浏览器实例。

## 配置文件解析

PinchTab 使用以下优先级确定 Chrome 的 `--user-data-dir`（配置文件）：

1. **显式 `ProfileDir`**：如果在配置或作为标志提供了特定路径，PinchTab 使用该确切目录。
2. **命名配置文件**：如果提供了配置文件名称（例如，通过仪表板或 CLI），PinchTab 将其解析为 `ProfilesBaseDir` 内的子目录。
3. **默认配置文件**：如果未指定配置文件，默认为 `~/.pinchtab/profiles/default`（在 Linux/macOS 上）。

## 单例模型

Chrome 对每个用户数据目录强制执行**单例**模型。这意味着：
* 在任何给定时间，只有**一个** Chrome 进程可以使用特定的配置文件目录。
* 如果第二个进程尝试使用同一个目录，Chrome 要么启动失败，要么（在某些配置中）尝试在现有进程中打开一个新窗口。

PinchTab 通过在配置文件目录中使用 `pinchtab.pid` 文件添加额外的保护层，确保只有一个 PinchTab 实例管理特定的配置文件。

## 新无头模式和并行实例

随着**新无头模式**（`--headless=new`）的引入，Chrome 的配置文件共享行为变得更加严格：
* **无共享**：您不能在多个并发实例之间重用相同的 `--user-data-dir`。
* **需要单独目录**：并行浏览器**必须**使用单独的目录，以避免随机启动错误和锁冲突。

### 无头自动回退
为了支持并行自动化任务，PinchTab 为无头实例实现了**自动回退**：
1. 如果无头实例尝试使用已被另一个 PinchTab 进程锁定的配置文件，它将**自动创建一个唯一的临时目录**（例如，`/tmp/pinchtab-profile-*`）。
2. 这允许您并行运行多个无头任务，而无需手动管理配置文件路径。

### 手动并行（有头模式）
在**有头模式**下，PinchTab 不会自动回退到临时目录（以避免意外丢失用户会话数据）。如果您需要并行运行多个有头浏览器，必须：
* 使用不同的命名配置文件。
* 为每个实例明确提供唯一的 `--user-data-dir`。

## AI 代理的最佳实践

在构建使用 PinchTab 的代理时，请遵循以下指南：

* **持久性**：如果您需要浏览器在会话之间记住登录、Cookie 或历史记录，请使用命名配置文件（例如，`agent-alpha`、`agent-beta`）。
* **隔离**：对于一次性任务或高并发抓取，依赖默认的无头模式，该模式在发生冲突时自动处理目录隔离。
* **清理**：如果您手动创建临时目录，请确保在任务完成后清理它们，以避免填满磁盘空间。

## 故障排除

如果您看到错误消息 "The profile appears to be in use by another Chromium process"：
1. **检查活动实例**：确保没有其他 PinchTab 或 Chrome 进程已经在使用该配置文件。
2. **过时的锁**：如果没有活动进程，PinchTab 将在下一次启动时尝试自动清除过时的 `SingletonLock` 文件。
3. **手动修复**：在极少数情况下，您可能需要手动从配置文件目录中删除 `SingletonLock` 文件。

有关 PinchTab 如何从崩溃中恢复的更多详细信息，请参阅 [Chrome 配置文件锁恢复](chrome-profile-lock-recovery.md)。
