# 基准测试：深度分析

本文档比较了两个代理驱动的浏览器表面 — **PinchTab** 和 **agent-browser** — 在共享基准测试任务集上的表现，通过我们的 API 运行器驱动。目标是测量使用每个工具与 LLM 代理的真实端到端令牌成本，而不是抽象的 "工具成本"。

主要锚点是在 `claude-haiku-4-5-20251001` 上进行的 10 步基本范围比较，每个通道 n=5（`lp1`–`lp5`，`la1`–`la5`）。它由同一模型上的 24 步扩展范围比较（每个通道 n=3，`lpe1`–`lpe3` / `lae1`–`lae3`）和 `claude-sonnet-4-6` 上的 24 步扩展范围比较（每个通道 n=2，`lpe-sonnet46-*` / `lae*-sonnet46-*`）作为补充。所有数据都记录在 2026-04-20。原始日志附加在本文档底部。

## 摘要

在我们测量的每个范围内，PinchTab 都比 agent-browser 更便宜且使用更少的 API 往返。下面的百分比是 *相对于 agent-browser*（即 "PinchTab 在该指标上比 agent-browser 便宜 N%"）。

| 范围 | 模型 | n | PinchTab 成本 | agent-browser 成本 | PinchTab 更便宜 | 更少请求 | 更少令牌 |
|-------|-------|--:|--------------:|-------------------:|-----------------:|---------------:|-------------:|
| 基本（10 步）    | Haiku 4.5  | 5 | $0.1024 | $0.1132 |  **9.5%** | 23.0% | 17.9% |
| 扩展（24 步） | Haiku 4.5  | 3 | $0.3516 | $0.4372 | **19.6%** | 31.1% | 26.2% |
| 扩展（24 步） | Sonnet 4.6 | 2 | $0.8932 | $1.1204 | **20.3%** | 29.4% | 25.3% |

- **成本差距在更长范围内扩大**（Haiku 上从 9.5% → 19.6%）。点击→快照往返随着步数增加而复合。
- **通道差距在扩展范围内大致与模型无关**（Haiku 19.6%，Sonnet 20.3%）。更强的模型不会通过推理绕过额外的往返。
- **请求减少多于令牌减少多于成本减少**。agent-browser 的额外令牌大部分是 `cache_read`，价格为 $0.10/1M（Haiku）或 $0.30/1M（Sonnet）— 每个令牌便宜但数量很多。

## 测试如何进行

两个通道完成 **相同的 10 步** — 组 0 中的 5 步（导航、快照、读取文本、点击链接、提取特定数据）加上组 1 中的 5 步（填写文本输入、点击按钮、使用搜索功能、处理动态内容、导航返回）。两个通道读取 **共享任务文件**，无通道特定任务覆盖。

### 环境

每次运行都在 Docker Compose（`tests/tools/docker-compose.yml`）中执行，包含三个服务：

- `fixtures` — 托管测试页面的基准 Web 服务器（`/`、`/wiki.html`、`/articles.html`、`/search.html`、`/form.html`、`/dashboard.html`、`/ecommerce.html`、`/spa.html`、`/login.html` 等）。
- `pinchtab` 或 `agent-browser` — 正在测量的浏览器表面。PinchTab 由 `tests/tools/config/pinchtab-benchmark.json` 构建（IDPI `wrapContent=false`，以匹配 agent-browser 的未包装输出）。
- `runner` — `tests/tools/runner/` 处的 Go 程序，驱动 LLM 代理循环。

### 代理循环

对于每个步骤：

1. 运行器将任务描述加上累积的转录发送给 Anthropic。
2. 模型发出 shell 命令（`./scripts/pt ...`、`./scripts/ab ...` 或 `./scripts/runner step-end ...` 之一）。
3. 运行器在通道的容器内执行命令，并将结果附加到转录中。
4. 模型回答后，`./scripts/runner step-end` 记录答案并根据预言机验证它。
5. 循环继续，直到所有步骤都被回答或达到 `--max-turns`。

令牌使用量直接从 Anthropic 的每个响应的 `usage` 对象读取，并在整个运行中求和 — 模型不进行自我报告。

### 技能的作用

两个通道都提供了一个 "技能" — 一个 Markdown 包，教模型如何驱动工具：命令形状、引用语法、快照格式、已知陷阱、恢复模式。该技能位于每个请求的 **缓存前缀** 中，按 `cache_read` 费率收费（Anthropic 上未缓存输入的 10%），因此无论长度如何，每轮费用都可预测且很小。

在 10 步短运行中，技能做了三件重要的事情：消除探索轮次（没有技能，代理必须戳 `--help`、打印 `README.md`、尝试错误标志并恢复），锚定引用语法（`e5`、`@e5` 或 `[~]` 标记的含义），并预先处理已知陷阱（点击时的导航 409，PinchTab 上的 `--snap-diff` 格式；agent-browser 上的 `@ref` 前缀和会话语义）。

技能不做工作；它让代理用更少的轮次做工作。令牌节省几乎完全来自轮次计数减少。

### 为什么我们使用 *部分* 技能运行

两个通道都针对完整技能包的 **修剪子集** 运行，而不是产品在日常使用中会加载的完整技能包。这是故意的，以保持比较公平：

- **PinchTab 完整技能** = `skills/pinchtab/SKILL.md`（~14.5 KB）+ `skills/pinchtab/references/` 下的六个参考文件（api.md、commands.md、env.md、mcp.md、profiles.md、agent-optimization.md）总计 ~44 KB。完整大小 ≈ **58.5 KB**。
- **基准测试中的 PinchTab** = 仅 `SKILL.md`（~14.5 KB）。参考子文件夹未内联。
- **agent-browser 完整技能** 来自 `agent-browwser skills get agent-browser --full`。在基准测试中，运行器仅提取标题加上 `references/commands.md` 和 `references/snapshot-refs.md`，并丢弃其余部分（参见 `tests/tools/runner/prompt.go:DownloadAgentBrowserSkill`）。

在 10 步测试中，完整的参考捆绑包会在恰好提供更多参考内容的通道上主导 `cache_read` 令牌。让两个通道为代理在 10 步运行中从未查阅的内容付费，会放大编写更多文档的通道，而不是在实际任务中更高效的工具。将两者都修剪为 "标题 + 代理实际使用的一个参考文件" 隔离了工具表面比较。

如果您想测量 *日常生产使用中的成本* 而不是 *工具表面本身的成本*，请在两个通道上使用 `--full` 技能重新运行。

方差仍然足够高，因此单次运行的结论应谨慎对待；n=5 给出了可用的中心趋势，但置信区间很宽。

### 差距如何随工作负载扩展

| 范围 | 成本/步（pt） | 成本/步（la） | PinchTab 更便宜 |
|-------|---------------:|---------------:|-----------------:|
| 基本（10 步，n=5） | $0.01024 | $0.01132 | **9.5%** |
| 扩展（24 步，n=3） | $0.01465 | $0.01822 | **19.6%** |

有两件事在变化：

1. **差距在更长范围内扩大**。PinchTab 在 10 步上便宜约 9.5%；在 24 步上便宜约 19.6%。点击→快照乒乓模式复合 — 每次涉及动作后快照的额外步骤都会让 agent-browser 付出额外的往返，而 PinchTab 通过 `--snap-diff` 将其打包为一次。
2. **两个通道的每步成本都上升**，因为添加了组 2–5（pt $0.0102 → $0.0147/步，la $0.0113 → $0.0182/步）。后面的组包含结构上更难的步骤 — 具有动态状态的仪表板、电子商务流程、SPA 交互 — 在任一表面上都需要更多轮次。

**agent-browser 在扩展范围内的方差更大**。lae3（160 请求，$0.5132）是一个明显的异常值；没有它，la 均值降至 $0.3992，PinchTab 的优势缩小至 ~12%。PinchTab 的 lpe 运行保持在 $0.329–$0.365 的狭窄范围内。

### 结论

扩展范围运行强化了基本范围的故事，但规模更大：在生产现实的工作负载下，PinchTab 明显更便宜（约五分之一更少）并且使用显著更少的 API 往返（约 31% 更少）。基本套件的 9.5% 数字低估了规模上的差距。

对于标题引用：使用基本数字作为最小噪声的苹果对苹果锚点（紧凑组，高重复计数），并在谈论现实工作负载大小的成本时引用扩展数字。

### 结论

PinchTab 的 **~20% 成本优势在两种模型的扩展范围内基本相同**（Haiku 19.6%，Sonnet 20.3%）。更强的推理不会消除点击→快照乒乓；额外的往返是工具表面的结构特性，而不是模型通过更好的规划修复的故障模式。

两个次要观察：

- **Sonnet 在两个通道上使用的请求略少于 Haiku**（lpe 87.5 vs 92.3，lae 124.0 vs 134.0）— 更强模型的 ~5–7% 轮次减少。令牌总数类似下降（~9–10%）。对比较影响不大。
- **Sonnet 在相同工作负载上的成本约为 Haiku 的 2.5 倍**（lpe $0.89 vs $0.35；lae $1.12 vs $0.44），这与 ~3× 价格比因轮次减少而略有折扣一致。

两次运行很薄弱 — n=2 具有非常宽的置信区间，我们看到 lpe 上 ~7% 的通道内差异和 lae 上 ~6% 的差异，因此不要对 20.3% 数字读取精度。方向故事（优势在模型变化中存活）是承载结果。

## 公平性警告

### 1. 任务套件偏差

10 步任务集是在 PinchTab 开发过程中设计的。在 agent-browser 中笨拙或多调用的步骤 — 那些需要在点击 / 填充 / 提交 / 后退 / 动态内容交互后显式动作后快照的步骤 — 构成了组 1 的大部分和组 2–5 的全部。

更强的未来比较将与两个团队共同设计任务集，或运行更大的任务集（完整的 39 组基准测试位于 `tests/benchmark/` 中），以便每个任务的特殊偏差平均化。

### 2. 部分技能配置

如上所述，两个通道都针对其完整技能的匹配 *子集* 运行。这对于测量工具表面是公平的，但不代表生产使用。不同的技能配置可能对 agent-browser 有所帮助。

## 方法说明

### 如何测量令牌使用

运行器直接从 Anthropic 的每个响应的 `usage` 对象读取使用情况，并在整个运行中求和。每个响应捕获的字段：

- `input_tokens` — 新鲜未缓存输入
- `cache_creation_input_tokens` — 提示缓存写入
- `cache_read_input_tokens` — 提示缓存命中
- `output_tokens` — 输出
- `request_count` — 运行的 API 请求

### 令牌总数代表什么

这是 **整个代理循环** 的成本 — 系统提示、技能、工具调用、工具输出、推理和重试 — 而不是纯粹的 "仅浏览器工具" 数字。它不包括 Docker CPU 时间或本地 shell 执行成本。

### 上下文压缩

为防止运行器在每轮重新发送整个对话（天真默认），我们：

1. 在将工具输出反馈给模型之前截断它（每个调用 2 行预览）。
2. 将旧历史压缩为从基准报告派生的简短进度摘要。
3. 将通道设置 + 技能内联到缓存前缀中，以便代理不会花费未缓存轮次运行 `cat setup-*.md`。

### 步骤结束折叠

每个完成的步骤过去需要两个簿记轮次：一个记录答案，一个根据预言机验证它。这些被折叠成单个 `./scripts/runner step-end` 调用。上面的 n=5 运行确认两个通道都 10/10 采用；与早期基线相比，这 alone 为每个 PinchTab 运行节省约 10 轮，为每个 agent-browser 运行节省约 13 轮。

## 重现此基准

```bash
# 从仓库根目录：

# 1. 基线通道（确定性，~30 秒）
./dev opt baseline

# 2. PinchTab 通道（需要 Anthropic 密钥）

# 3. agent-browser 通道（需要 Anthropic 密钥）

jq '.run_usage' tests/benchmark/results/agent_browser_benchmark_*.json
```

- 分数是通过计数，不是答案质量或完成时间
